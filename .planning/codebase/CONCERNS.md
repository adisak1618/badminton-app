# Codebase Concerns

**Analysis Date:** 2026-04-05

## Project Status

This is a fresh Turborepo monorepo scaffold (created via `create-turbo`). The codebase is in a minimal state with boilerplate content. No production code or significant business logic currently exists.

## Test Coverage Gaps

**No Testing Infrastructure:**
- What's not tested: No test framework or tests exist
- Files: All files - `apps/web/`, `apps/docs/`, `packages/ui/`
- Risk: New features cannot be validated before merge. Regressions may not be caught.
- Priority: High
- Note: No Jest, Vitest, or other test runner configured in any package

**UI Component Library Not Tested:**
- What's not tested: Button, Card, Code components in `packages/ui/src/`
- Files: `packages/ui/src/button.tsx`, `packages/ui/src/card.tsx`, `packages/ui/src/code.tsx`
- Risk: Component API changes will not be caught when consumed by apps
- Priority: High

## Unused Exported Components

**Card and Code Components:**
- Issue: Card and Code components exported from `packages/ui/src/` but never imported in actual app pages
- Files: `packages/ui/src/card.tsx`, `packages/ui/src/code.tsx`
- Impact: Dead code - increases package size and creates confusion about what's actually used
- Fix approach: Either use these components in the apps or remove them from the library

## Type Safety Issues

**Incomplete TypeScript Configuration:**
- Issue: TypeScript is configured but not all edge cases are covered
- Files: `apps/web/tsconfig.json`, `apps/docs/tsconfig.json`
- Current status: `strictNullChecks: true` enabled but other strict options disabled
- Impact: Potential runtime errors from unhandled null/undefined values
- Recommendations: Enable `noImplicitAny`, `noFallthroughCasesInSwitch`, `noImplicitReturns`, `noUnusedLocals`, `noUnusedParameters`

## Linting Configuration Weakening

**ESLint Configured with `onlyWarn` Plugin:**
- Issue: All ESLint errors converted to warnings via `eslint-plugin-only-warn`
- Files: `packages/eslint-config/base.js` (lines 5, 24-27)
- Impact: Linting errors won't block builds, PRs, or deploys. Code quality issues accumulate
- Fix approach: Remove `eslint-plugin-only-warn` and enforce strict linting. Fix violations in existing code instead
- Current risk: New team members may ignore warnings thinking they're optional

**No Lint Enforcement on Build:**
- Issue: ESLint runs with `--max-warnings 0` in CI but doesn't fail the build in a documented way
- Files: `apps/web/package.json` (line 10), `packages/ui/package.json` (line 9)
- Impact: Developers can commit code that would fail CI lint checks
- Recommendations: Add pre-commit hooks (husky) to catch violations before commit

## Architecture Fragility

**Hardcoded Turborepo Bootstrap Content:**
- Issue: Both `apps/web` and `apps/docs` contain identical boilerplate from template
- Files: `apps/web/app/page.tsx`, `apps/docs/app/page.tsx`
- Problem: 102 lines each of duplicate code that references "Create Next App" and Vercel templates
- Impact: Difficult to differentiate apps; confusing for new developers
- Fix approach: Replace with actual application content and unique landing pages

**Metadata Not Customized:**
- Issue: Metadata hardcoded with template values
- Files: `apps/web/app/layout.tsx` (line 15), `apps/docs/app/layout.tsx` (line 15)
- Current: `title: "Create Next App"`, `description: "Generated by create next app"`
- Impact: SEO, browser tabs, and documentation all show placeholder values
- Fix approach: Set meaningful, per-app metadata

## Missing Project Context

**No Clear Documentation:**
- What's missing: README doesn't explain what "badminton" project does
- Files: `README.md`
- Current state: Generic Turborepo starter documentation with no project-specific guidance
- Risk: New developers don't understand project purpose or architecture
- Recommendations: Add project charter, high-level architecture diagram, and onboarding guide

**No .env Setup:**
- Issue: `.npmrc` exists but is empty
- Files: `/Users/adisakchaiyakul/project/badminton/.npmrc`

## Monorepo Maintenance Issues

**Version Consistency Gaps:**
- Issue: React version caret (`^19.2.0`) allows minor/patch updates; other deps are exact
- Files: `apps/web/package.json`, `apps/docs/package.json`, `packages/ui/package.json`
- Risk: Inconsistent React versions across workspace if pnpm install runs with different versions
- Fix approach: Use same pinning strategy across all packages (recommend all exact versions)

**Node Version Not Documented:**
- Issue: package.json specifies `"engines": { "node": ">=18" }`
- Files: `/Users/adisakchaiyakul/project/badminton/package.json` (line 18)
- Risk: >=18 is very broad. Codebase may have been tested on specific version only
- Recommendations: Document actual tested version (e.g., 20.x, 22.x) and add `.nvmrc`

**Missing Workspace Scripts:**
- Issue: Root package.json has build/dev/lint but no clean, format, or pre-commit hooks
- Files: `/Users/adisakchaiyakul/project/badminton/package.json`
- Impact: No consistent way to clean generated files, format code, or prevent bad commits
- Recommendations: Add `clean`, `precommit`, `postinstall` scripts

## Build and Deployment Concerns

**Empty Next.js Config:**
- Issue: Both Next.js apps have empty configuration
- Files: `apps/web/next.config.js`, `apps/docs/next.config.js`
- Current: `const nextConfig = {}; export default nextConfig;`
- Impact: No build customization, no image optimization config, no rewrites/redirects
- Risk: Default Next.js behavior may not match requirements when more features added

**Build Output Caching:**
- Issue: Turbo task caching only documented for build, not for other tasks
- Files: `turbo.json` (lines 5-8)
- Current: Build outputs `.next/**` but caches are mixed with source
- Impact: Build cache invalidation unclear; CI may use stale caches
- Recommendations: Document cache strategy for each task

## Code Quality Observations

**Lack of Error Boundaries:**
- Issue: React apps have no error boundary components
- Files: `apps/web/app/layout.tsx`, `apps/docs/app/layout.tsx`
- Risk: Any unhandled component error crashes entire app with no user-friendly fallback
- Recommendations: Add error.tsx and global-error.tsx per Next.js guidelines

**No Loading States:**
- Issue: No loading.tsx files or Suspense boundaries defined
- Files: `apps/web/app/`, `apps/docs/app/`
- Impact: Poor UX during slow operations or navigation
- Recommendations: Add proper skeleton screens and loading indicators

**Client Component Usage Without Verification:**
- Issue: Button component uses `"use client"` directive but never verified to use only client-safe APIs
- Files: `packages/ui/src/button.tsx` (line 1)
- Risk: May break if server-only dependencies accidentally imported
- Recommendations: Add tests validating client-safety

## Potential Security Gaps

**No Security Headers Configuration:**
- Issue: Next.js security headers (CSP, X-Frame-Options, etc.) not configured
- Files: `apps/web/next.config.js`, `apps/docs/next.config.js`
- Impact: No protection against XSS, clickjacking, or MIME type sniffing
- Recommendations: Add next-secure-headers or configure headers in next.config.js

**External Link Security:**
- Issue: Links in Card component and page templates use `target="_blank"` without `noreferrer`
- Files: `packages/ui/src/card.tsx` (line 18)
- Current: Has `rel="noopener noreferrer"` - this is correct
- Note: Page templates also correctly include this - no issue here

**No CSRF Protection Strategy:**
- Issue: If this becomes an API-serving app, no CSRF tokens or protection mentioned
- Files: None yet, but will matter when server actions added
- Risk: Future form submissions vulnerable to CSRF
- Recommendations: Document CSRF strategy when server actions planned

## Performance Concerns

**Font Loading Not Optimized:**
- Issue: Local fonts loaded synchronously
- Files: `apps/web/app/layout.tsx` (lines 5-12), `apps/docs/app/layout.tsx`
- Current: Using `localFont()` with `.woff` files
- Impact: Blocks initial render; no font-display CSS specified
- Recommendations: Add `display: 'swap'` to font config

**Image Component Without Proper Configuration:**
- Issue: Next.js Image components used but not all optimization settings configured
- Files: `apps/web/app/page.tsx` (multiple), `apps/docs/app/page.tsx`
- Current: Uses `priority` on one image but unclear what happens to others
- Recommendations: Define image widths/heights and use `sizes` attribute

**No CSS Optimization:**
- Issue: CSS modules used but no global style deduplication strategy
- Files: `apps/web/app/page.module.css`, `apps/docs/app/page.module.css`
- Impact: Potential duplicate CSS shipped to browser
- Recommendations: Add CSS-in-JS or Tailwind for consistency

## Fragile Coupling

**Tight Coupling to Turborepo Template:**
- Issue: Code heavily references "Turborepo", "Next.js", and Vercel examples
- Files: All page files and README
- Impact: Makes it hard to pivot away from Turborepo/Vercel if needed
- Recommendations: Replace template boilerplate with actual domain logic

**Unclear Package Boundaries:**
- Issue: UI package exports all components with glob pattern `"./*": "./src/*.tsx"`
- Files: `packages/ui/package.json` (lines 6)
- Problem: All components always exported even if not ready; no way to mark as private
- Impact: Consumer apps can import any component without clear API contract
- Recommendations: Use explicit exports with semantic versioning

## Type Definition Issues

**Missing JSX Element Return Type in Some Components:**
- Issue: Code inconsistently uses JSX.Element vs ReactNode for returns
- Files: `packages/ui/src/card.tsx` (line 13), `packages/ui/src/code.tsx` (line 9)
- Current: These specify JSX.Element correctly but Button doesn't
- Impact: Inconsistent type contracts make it unclear what's expected
- Recommendations: Standardize all components to return JSX.Element
