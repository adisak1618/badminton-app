1. Create `apps/api/src/lib/idempotency.ts` — the idempotency helper that uses `onConflictDoNothing`:
```typescript
import { db, idempotencyKeys } from "@repo/db";

/**
 * Attempts to record a webhook event ID. If already recorded (duplicate delivery),
 * returns false. If newly recorded, returns true and the handler should process it.
 *
 * Uses PostgreSQL unique constraint — atomic and handles concurrent deliveries.
 */
export async function processWithIdempotency(
  webhookEventId: string,
  handler: () => Promise<void>
): Promise<void> {
  const result = await db
    .insert(idempotencyKeys)
    .values({ webhookEventId })
    .onConflictDoNothing()
    .returning();

  // result is empty array if conflict (already processed) — this is expected behavior
  if (result.length === 0) {
    return; // duplicate — skip silently
  }

  await handler();
}
```

2. Create `apps/api/src/webhook/handlers/event-dispatcher.ts` — thin dispatcher for webhook events:
```typescript
import type { WebhookEvent } from "@line/bot-sdk";
import { processWithIdempotency } from "../../lib/idempotency";

/**
 * Dispatches a single webhook event through the idempotency gate.
 * Each event type will be handled by its own handler in future phases.
 * For Phase 1, we only need the idempotency check — no business logic yet.
 */
export async function dispatchEvent(event: WebhookEvent): Promise<void> {
  // webhookEventId is available on all webhook events
  const eventId = event.webhookEventId;
  if (!eventId) {
    // Events without an ID cannot be deduplicated — process directly
    // This should not happen with LINE Messaging API v2, but handle gracefully
    return;
  }

  await processWithIdempotency(eventId, async () => {
    // Phase 1: log the event type for debugging; actual handlers added in Phase 2+
    console.log(`[webhook] Processing event: ${event.type} (${eventId})`);
  });
}
```

3. Create `apps/api/src/webhook/line.ts` — the Elysia webhook route. CRITICAL: Do NOT declare a `body` schema on this route. Access the raw body via `request.text()` BEFORE any JSON parsing:
```typescript
import { Elysia } from "elysia";
import { validateSignature } from "@line/bot-sdk";
import type { WebhookRequestBody } from "@line/bot-sdk";
import { env } from "../env";
import { dispatchEvent } from "./handlers/event-dispatcher";

export const lineWebhook = new Elysia()
  .post("/webhook/line", async ({ request, set }) => {
    // MUST call request.text() BEFORE any JSON parsing
    // Do NOT declare body schema on this route — it would consume the stream
    const rawBody = await request.text();
    const signature = request.headers.get("x-line-signature") ?? "";

    if (!validateSignature(rawBody, env.LINE_CHANNEL_SECRET, signature)) {
      set.status = 401;
      return "Unauthorized";
    }

    const payload = JSON.parse(rawBody) as WebhookRequestBody;

    // Return 200 immediately — dispatch events asynchronously
    // LINE retries if the response is delayed
    void Promise.all(
      payload.events.map((event) => dispatchEvent(event))
    );

    return "OK";
  });
```

Key design decisions:
- No body schema declaration (Elysia would consume the stream — see Research Pattern 1)
- `request.text()` called first to get raw bytes for signature verification
- `validateSignature` from `@line/bot-sdk` handles timing-safe HMAC-SHA256 comparison
- Return "OK" immediately; event processing is fire-and-forget via `void Promise.all`
- Each event goes through `dispatchEvent` which runs the idempotency check

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet -> /webhook/line | Untrusted HTTP POST from Line platform (or attacker) crosses into API |
| API -> Database | Idempotency inserts cross into shared PostgreSQL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Spoofing | POST /webhook/line | mitigate | `validateSignature(rawBody, channelSecret, signature)` from `@line/bot-sdk` — HMAC-SHA256 with timing-safe comparison; returns 401 on failure |
| T-01-05 | Tampering | Request body | mitigate | `request.text()` reads raw bytes BEFORE any parsing; signature computed against exact raw bytes; no body schema declared to prevent Elysia stream consumption |
| T-01-06 | Tampering | Webhook replay | mitigate | `idempotency_keys` table with `onConflictDoNothing()` — PostgreSQL unique constraint is atomic; handles concurrent duplicate deliveries |
| T-01-07 | Denial of Service | Missing env vars | mitigate | `@t3-oss/env-core` validates LINE_CHANNEL_SECRET, LINE_CHANNEL_ACCESS_TOKEN, DATABASE_URL at startup; crashes with descriptive error before serving any request |
| T-01-08 | Information Disclosure | Error responses | mitigate | Webhook returns only "Unauthorized" (401) or "OK" (200) — no stack traces or internal state leaked in responses |
| T-01-09 | Elevation of Privilege | Forged webhook creating DB records | mitigate | Signature verification gate runs before any DB operation; forged requests never reach dispatchEvent |
</threat_model>
